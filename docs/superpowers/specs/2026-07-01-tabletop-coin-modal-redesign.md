# 桌面铜钱悬浮弹窗重设计

日期：2026-07-01
状态：待用户复核

## 1. 设计结论

本次重设计将应用改成一个极简的实体桌面/台面场景。主画面不再承担表单、说明、结果卡片或应用窗口职责；常态下只呈现桌面环境和三枚 3D 铜钱。

所有交互和信息展示都通过最上层悬浮弹窗完成，包括 AI API 配置、所问之事、投掷过程提示、AI 解读、原始卦象、起卦过程和传统依据。

这里的“桌面”指物理桌面、案台、木桌、石台或宣纸铺面，不是桌面应用窗口，也不是带标题栏、侧栏、工具栏的系统窗口。

## 2. 产品目标

- 让用户进入页面后直接感受到一个安静、聚焦的起卦台面。
- 让三枚 3D 铜钱成为主画面唯一核心对象。
- 让六次投掷具有真实重量感、碰撞感和停稳反馈。
- 让 AI 配置、问题输入、解读结果和原始卦象全部以悬浮弹窗呈现。
- 保留现有三枚铜钱起卦规则、本卦、动爻、变卦和传统依据。
- 解读完全走 AI；不保留任何本地白话解读兜底。

## 3. 非目标

- 不做桌面应用壳、窗口标题栏、仿 macOS/Windows 应用界面。
- 不在主画面放固定侧栏、卡片、结果面板、输入框或说明区。
- 不引入账号、历史记录、分享、支付或后端数据库。
- 不让摄像头手势决定随机结果；随机性仍由浏览器加密随机数生成。
- 不在 AI 请求失败时生成本地解释文本。

## 4. 现有代码基线

当前代码已经具备可复用的规则和 AI 基础：

- `src/domain/coinToss.ts`：三枚铜钱计分、六爻生成和加密随机。
- `src/domain/interpretation.ts`：本卦、动爻、变卦和传统依据组装。
- `src/ai/openaiReading.ts`：OpenAI-compatible、Anthropic、DeepSeek 请求和 JSON 解析。
- `src/hooks/useCastingSession.ts`：`question`、`casting`、`result` 三阶段起卦状态。

主要需要重做的是呈现层：

- `src/App.tsx`：从阶段式整页切换改成桌面场景 + 全局弹窗调度。
- `src/components/QuestionEntry.tsx`：拆成 AI 配置弹窗和问题弹窗。
- `src/components/CastingStage.tsx`：改成纯桌面投掷场景。
- `src/components/CoinAnimation.tsx`：从文字圆片改成 3D 铜钱动画。
- `src/components/ResultView.tsx`：改成结果弹窗内容，而不是页面栏布局。
- `src/styles.css`：从网页卡片布局改成全屏桌面场景和弹窗系统样式。

## 5. 用户流程

1. 用户打开页面，看到一张空的实体桌面和三枚静置铜钱。
2. 如果 AI API 信息不完整，页面立即显示 `AI 配置`悬浮弹窗，背景桌面保持可见但不可操作。
3. 用户补全 Provider、API URL、API Key 和模型后关闭配置弹窗。
4. 页面显示 `所问之事`悬浮弹窗，用户输入问题或选择快捷问题。
5. 用户确认后弹窗关闭，主画面只剩桌面和三枚铜钱。
6. 用户通过手势、键盘或直接点击/触摸铜钱触发每一掷；三枚铜钱在桌面上抛起、翻转、碰撞、弹跳并停稳。
7. 每次停稳后记录一爻，并用短暂轻浮层提示当前进度，例如“第 3 掷 / 共 6 掷”。
8. 六次完成后，页面请求 AI 解读，并打开 `AI 解读`悬浮弹窗。
9. AI 解读成功后，弹窗展示 AI 白话解读和行动建议。
10. 同一个结果弹窗内保留 `原始卦象`、`起卦过程`、`传统依据`页签。
11. AI 解读失败时，弹窗只展示失败原因、重试和修改 API 入口，不生成本地兜底解读。

## 6. 主画面设计

主画面是全屏物理桌面场景。

桌面可以采用深色木纹、石台或宣纸铺面，但必须克制。它的职责是承托铜钱，不承载信息密度。主画面中不出现固定说明文案、表单、卦象卡片、面板、导航或应用窗口。

默认可见元素：

- 三枚 3D 铜钱。
- 投掷时的铜钱运动和桌面阴影。
- 必要时短暂出现的轻量浮层，例如进度、错误或“点击继续”。

主画面应避免：

- 大面积网格背景。
- 仪表盘式布局。
- 多卡片并排。
- 长文案直接铺在页面。
- 把弹窗内容伪装成固定页面区域。

## 7. 悬浮弹窗系统

所有弹窗都应位于最高交互层，并带背景遮罩或景深虚化。弹窗视觉应与桌面场景协调，但仍保持高可读性。

弹窗类型：

- `AiSettingsDialog`：API 配置。首次缺配置时强制显示；用户可在结果失败后再次打开。
- `QuestionDialog`：所问之事。支持快捷问题和自定义问题。
- `CastProgressToast`：短暂进度提示。只显示当前第几掷、是否已记爻，不常驻为面板。
- `ResultDialog`：AI 解读结果。包含 AI 正文、建议、原始卦象、起卦过程和传统依据。
- `ErrorDialog`：阻塞性错误，例如 AI 请求失败且用户需要重试或修改配置。

弹窗行为：

- API 配置不完整时，不能开始起卦。
- 问题为空时，不能开始起卦。
- 结果弹窗可以关闭，但关闭后主画面仍不显示固定结果卡片或独立浮动按钮。
- 结果弹窗关闭后，用户只能通过点击/触摸铜钱或键盘快捷操作重新打开结果弹窗；主画面仍保持只有桌面和铜钱。

## 8. 3D 铜钱设计

铜钱必须呈现为 3D 对象，而不是 CSS 圆片或文字图标。

外观要求：

- 圆形铜钱，中心方孔。
- 有厚度、倒角、边缘磨损和铜锈质感。
- 正反两面有可区分纹理或文字纹样。
- 光照下有金属粗糙度变化，不应像扁平按钮。

实现建议：

- 使用 Three.js 或 React Three Fiber 渲染铜钱和桌面。
- 铜钱几何可以由圆柱体减方孔、导入轻量模型，或用可控几何体组合实现。
- 材质使用铜色金属材质、粗糙度贴图或程序化噪声。
- 桌面使用平面几何、纹理和接触阴影。

## 9. 投掷动画

每次投掷分为五段：

1. 起手：三枚铜钱从桌面轻微抬起。
2. 抛掷：沿不同弧线向上和向前运动。
3. 翻转：每枚铜钱以不同角速度绕多个轴旋转。
4. 碰撞：触桌后产生一次或两次短弹跳，并伴随滑动或滚动。
5. 停稳：动画以预先生成的随机结果作为最终朝向。

规则要求：

- 最终正反面必须由现有 `tossCoins()` 结果决定。
- 动画不能在视觉上暗示用户可以影响随机概率。
- 动画播放期间不能重复记爻。
- 动画结束后再提交该次投掷进入六爻记录。
- 移动端应降低粒子、阴影和弹跳复杂度，但保留翻转与停稳。

## 10. AI 解读

AI 解读是唯一白话解释来源。

保留现有原则：

- 发送用户问题、本卦、动爻、变卦、卦辞、象辞、爻辞和起卦过程。
- AI 返回 `headline`、`plainText` 和 `advice`。
- AI 不得改写传统原文。
- AI 不得给出医疗、法律、投资等高风险确定性建议。

失败处理：

- 网络失败、模型错误、Key 错误、格式错误时，只显示错误。
- 可以提供 `重试` 和 `修改 API 配置`。
- 不显示“白话解读”或“行动建议”占位内容。
- 不用本地模板补写解释。

## 11. 原始卦象保留方式

原始卦象仍必须完整保留，但只能出现在结果弹窗中。

结果弹窗建议页签：

- `AI 解读`：标题、白话解读、行动建议、AI 状态。
- `原始卦象`：所问之事、本卦、变卦、动爻、六爻图。
- `起卦过程`：六次投掷、三枚正反面、总分。
- `传统依据`：本卦卦辞、本卦象辞、动爻爻辞、变卦卦辞或无动爻说明。

传统依据保持原文和事实，不混入本地白话解释。

## 12. 状态模型

建议将现有起卦状态和弹窗状态拆开。

起卦状态：

- `idle`：桌面静置，等待配置或问题。
- `ready`：已配置 API 且已有问题，等待投掷。
- `casting`：正在六次投掷。
- `animating`：本次铜钱动画播放中。
- `reading`：六爻完成，AI 请求中。
- `complete`：AI 成功或失败，结果可查看。

弹窗状态：

- `ai-settings`
- `question`
- `result`
- `error`
- `none`

这两个状态应独立管理，避免把弹窗开关硬编码进起卦规则。

## 13. 组件边界

推荐组件拆分：

- `TabletopScene`：全屏桌面和 3D 铜钱场景，只负责可视化和投掷动画。
- `CoinTossController`：把随机结果、动画完成回调和记爻动作串起来。
- `ModalLayer`：统一管理遮罩、焦点、关闭行为和层级。
- `AiSettingsDialog`：Provider、API URL、API Key、模型输入。
- `QuestionDialog`：快捷问题、自定义问题和开始起卦。
- `ResultDialog`：AI 解读和原始卦象页签。
- `HexagramFacts`：本卦、变卦、动爻、六爻图。
- `TraditionalBasis`：传统依据列表。

领域层保持纯函数，不依赖 3D 或 UI。

## 14. 可访问性与响应式

- 弹窗必须有明确标题、焦点陷阱、Esc 关闭规则和可键盘操作按钮。
- API 配置和问题弹窗属于阻塞流程时，背景桌面不能获得焦点。
- 铜钱动画应支持 `prefers-reduced-motion`，降低抛掷幅度和旋转速度。
- 移动端弹窗宽度使用视口约束，不出现横向滚动。
- 结果弹窗内容较长时，弹窗内部滚动，不让主画面滚动承担结果展示。

## 15. 测试策略

单元测试：

- 起卦规则、动爻、变卦和传统依据继续沿用现有测试。
- AI 请求失败时不渲染白话解读和行动建议。
- API 配置缺失时弹出配置弹窗并阻止开始。
- 问题为空时弹出问题弹窗并阻止开始。

组件测试：

- 初次打开且无 API Key 时出现 `AI 配置`弹窗。
- API 完整后出现 `所问之事`弹窗。
- 六次投掷后出现 `AI 解读`弹窗。
- 结果弹窗页签能显示原始卦象、起卦过程和传统依据。
- AI 失败时只出现错误、重试和修改配置入口。

视觉/交互验证：

- 桌面主画面没有固定表单、卡片、侧栏、工具栏或结果面板。
- 桌面常态只显示铜钱和桌面环境。
- 铜钱具备明显 3D 厚度、方孔和正反面差异。
- 投掷动画结束后的朝向与随机结果一致。
- 桌面和弹窗在桌面端、移动端都不遮挡核心操作。

## 16. 实施顺序

1. 建立全局弹窗状态和 API 配置弹窗。
2. 将问题输入从页面组件迁移到 `QuestionDialog`。
3. 改造 `App.tsx` 为 `TabletopScene` 常驻 + `ModalLayer` 覆盖。
4. 将 `ResultView` 改为 `ResultDialog`，用页签承载 AI 解读和原始卦象。
5. 引入 3D 铜钱场景，替换现有文字圆片铜钱。
6. 将投掷动作改成“生成随机结果 -> 播放动画 -> 动画完成后记爻”。
7. 更新样式和响应式规则，删除页面式卡片布局。
8. 更新测试覆盖新弹窗流程和无兜底 AI 失败路径。

## 17. 验收标准

- 打开页面时，如果 API 信息不完整，第一层交互是悬浮 `AI 配置`弹窗。
- 配置完成后，问题输入也以悬浮弹窗出现。
- 开始起卦后，主画面只剩桌面和三枚 3D 铜钱。
- 六次投掷期间，没有固定信息面板进入主画面。
- 铜钱投掷有明显三维翻转、落桌、弹跳和停稳效果。
- 六次完成后，AI 解读以悬浮弹窗显示。
- 本卦、变卦、动爻、六爻图、起卦过程和传统依据完整保留在弹窗内。
- AI 失败时不出现任何本地白话解读兜底。
