# 极致拟真 3D 掷钱六爻程序重设计

日期：2026-07-06
状态：待用户复核

## 1. 设计结论

本次重设计忽略当前体验实现，保留已验证的领域逻辑和 AI 适配能力。程序重做为一个物理优先、真实桌面感的 3D 掷钱起卦应用：

- 掷钱结果由三枚铜钱在 Rapier 物理世界中落定后的朝向决定。
- 玩家输入只生成物理初始条件，不直接生成正反面。
- PC 端使用“按住虚拟手心或钱筒摇动，松手抛出”的鼠标交互。
- 移动端使用加速度传感器“摇晃蓄势，静止自动释放”的交互。
- 六次掷钱从初爻到上爻生成传统六爻结果。
- 成卦后把用户问题、投掷记录、本卦、动爻、变卦和传统依据发送给用户配置的 OpenAI-compatible API 进行解读。
- 第一版保持纯前端，用户在浏览器中填写 API URL、API Key 和 Model。

视觉方向采用“博物馆级真实桌面”：真实铜钱、真实桌面材质、自然光照、弱 UI 和少量仪式细节。中心画面始终服务于铜钱物理运动，不用大面积玄学特效掩盖物理过程。

## 2. 目标

- 做出一个可感知为真实物件的 3D 掷钱程序，而不是预生成结果后的动画播放。
- 让玩家输入成为真随机初始条件的重要来源：鼠标轨迹、摇动节奏、手机加速度和旋转率都会进入物理初始状态。
- 严格遵循三枚铜钱六爻规则：每次三枚铜钱生成一爻，六次从下往上成卦。
- 结果可追溯：每一爻记录三枚正反面、分数、爻名、落定原因和输入摘要。
- AI 解读基于传统依据和用户问题，不改写经典原文，不做确定性断言。
- 桌面和移动端都有完整可用流程；传感器不可用时提供触控/键盘兜底。

## 3. 非目标

- 第一版不做后端代理、登录、账户、历史云同步或付费。
- 第一版不把 API Key 存到服务器；用户需理解前端直连会把 Key 暴露在浏览器环境中。
- 第一版不实现完整纳甲六爻体系，例如世应、六亲、六神、月建日辰和用神分析。
- 第一版不提供医疗、法律、投资等高风险确定性建议。
- 第一版不允许用随机数、AI 或动画目标直接决定铜钱正反面。

## 4. 推荐技术路线

在三种路线中选择物理优先方案：

1. 物理优先方案：输入生成物理初始条件，物理落定决定结果。拟真度和可信度最高，符合需求。
2. 视觉优先方案：优先做渲染和镜头，物理只保证看起来合理。上线快，但结果可信度不足。
3. 可审计优先方案：突出速度、能量和投掷证据。可信度强，但会牺牲沉浸感。

推荐路线的端到端链路固定为：

```text
玩家动作采样
-> 输入摘要
-> PhysicalTossInput
-> Rapier 三枚铜钱仿真
-> 落定刚体朝向读取 heads/tails
-> 三枚铜钱计分 6/7/8/9
-> 六次成卦
-> 卦象、爻辞、问题发给 AI 解读
```

## 5. 用户流程

主体验保持一个连续的 3D 桌面场景，不做传统 landing page。

1. 用户进入页面，看到真实桌面、三枚铜钱、问题入口和 AI 设置入口。
2. 用户填写问题并选择问题类型。
3. 用户确认 AI API URL、API Key 和 Model；也可以先完成传统结果，之后再补充 AI 配置重试解读。
4. 用户进入起卦状态。
5. PC 端按住钱筒或虚拟手心摇动鼠标；移动端摇晃手机。
6. PC 端松手释放；移动端达到能量阈值后，短暂停止自动释放。
7. 物理系统生成三枚铜钱初始状态并开始仿真。
8. 铜钱碰撞、翻转、滚动、落定。
9. 系统读取三枚铜钱最终朝向，记录一爻。
10. 用户重复六次，从初爻到上爻完成起卦。
11. 结果抽屉展示本卦、动爻、变卦、传统依据、投掷证据和 AI 解读状态。
12. 用户可重新起卦或查看每一爻投掷证据。

## 6. 状态机

应用状态机按起卦流程拆分：

```text
setup
ready
charging
released
simulating
settled
line-recorded
result
reading
reading-ready
reading-error
```

状态说明：

- `setup`：问题、问题类型和 AI 设置入口可见。
- `ready`：当前爻等待用户动作。
- `charging`：PC 钱筒摇动或移动端摇晃蓄势中。
- `released`：输入已封存，正在生成物理初始条件。
- `simulating`：Rapier 仿真运行，铜钱结果尚未读取。
- `settled`：三枚铜钱达到可读落定状态。
- `line-recorded`：当前爻写入 casting session。
- `result`：六爻完成，传统结果可见。
- `reading`：AI 解读请求进行中。
- `reading-ready`：AI 解读成功。
- `reading-error`：AI 请求失败，传统结果保留。

约束：

- 用户每一爻都必须完成一次动作；不提供一键生成六爻。
- `simulating` 前不允许展示当前爻结果。
- `settled` 前不允许调用六爻计分。
- AI 失败不影响传统结果展示和证据查看。

## 8. 输入设计

### 8.1 PC 鼠标钱筒

PC 端交互为“按住虚拟手心或钱筒摇动，松手抛出”。

采样规则：

- `pointerdown` 进入 `charging`。
- 持续记录最近约 800 到 1500 ms 的鼠标位置、时间戳、路径长度、曲率、速度变化和方向变化。
- UI 显示能量反馈，钱筒轻微跟随鼠标摆动，铜钱碰撞音效随能量增强。
- `pointerup` 封存输入，生成 `InputSummary`。

映射规则：

- 路径总长度影响总能量和初始线速度。
- 速度峰值影响抛出强度。
- 曲率和方向变化影响角速度。
- 持续时间影响三枚铜钱的相对偏移和散布。
- 释放瞬间方向影响主抛出方向。
- 微小手抖和采样摘要只参与物理扰动，不直接定面。

弱输入处理：

- 极弱摇动仍生成最小可投掷能量，避免程序卡住。
- 最小能量只影响速度和旋转，不改变最终面值。

### 8.2 移动端加速度摇晃

移动端交互为“摇晃蓄势，静止自动释放”。

流程：

- 用户授权 `DeviceMotionEvent`。
- 加速度或旋转率超过启动阈值，进入 `charging`。
- 摇晃期间累计能量、峰值次数、主加速度方向、旋转偏置和采样摘要。
- 达到有效能量后提示“静止以掷出”。
- 当能量低于停止阈值并持续一个安静窗口，自动释放。

映射规则：

- 总能量影响初始速度和角速度。
- 主加速度方向影响抛出方向。
- 旋转率变化影响三枚铜钱的 spin bias。
- 峰值次数影响三枚铜钱之间的差异和散布。
- 采样摘要与微扰种子只进入物理扰动。

兜底：

- 不支持传感器、权限被拒绝或采样异常时，移动端使用触控钱筒交互。
- 页面必须清楚展示权限状态，但不把权限失败做成阻断。

### 8.3 键盘兜底

键盘用于可访问性和测试：

- Space 或 Enter 按住进入蓄势。
- 按住时长和重复节奏生成标准轻投输入。
- 松开后走同一个 `PhysicalTossInput` 和物理仿真流程。

键盘兜底不得直接调用随机正反面。

## 9. 物理可信度设计

### 9.1 硬规则

- 正反面只从 Rapier 刚体最终朝向读取。
- 不允许提前生成 `heads` 或 `tails`。
- 不允许使用动画目标决定结果。
- 不允许超时后调用随机面值。
- AI 不参与起卦结果生成。

### 9.2 允许的扰动

为了模拟真实世界的不稳定性，可以使用输入摘要和微扰种子影响物理参数：

- 三枚铜钱初始位置的微小偏移。
- 初始姿态的微小差异。
- 初始角速度的微小差异。
- 摩擦和弹性的小范围变化。
- 桌面微倾或微小冲量。

这些扰动必须作用在物理变量上，不能直接映射为正反面。

### 9.3 随机性与可复现边界

运行时随机性的主要来源是玩家微动作、传感器噪声、采样时序和释放瞬间的人手差异。系统可以把这些输入压缩成摘要，用于生成物理初始条件和小幅扰动。

测试环境可以在给定完整 `PhysicalTossInput` 和固定微扰种子的情况下复现同一次仿真，以便验证物理和落定逻辑。这个可复现性只服务测试和调试，不改变运行时规则：正反面仍然只能由最终刚体朝向读取。

### 9.4 落定规则

严格落定 `strict`：

- 线速度低于阈值。
- 角速度低于阈值。
- 面法线接近向上或向下。

超时可读 `timeout-readable`：

- 仿真超过常规等待时间。
- 三枚铜钱的朝向都能读出正反面。
- 读取当前刚体朝向并记录落定原因为 `timeout-readable`。

立边处理 `edge-resolution`：

- 如果铜钱立边且不可读，不直接判面。
- 对不可读铜钱施加很小的物理扰动或重力偏置，让它继续自然倒下。
- 最终仍从刚体朝向读取正反面。

### 9.5 每爻证据

每一爻保存：

```text
throwIndex
inputSource
inputSummary
physicalTossInputDigest
settledReason
settledTimeMs
faces
score
lineName
isMoving
```

用户界面默认展示简洁记录；证据面板可展开查看详细输入摘要和落定原因。

## 10. 视觉与交互规格

### 10.1 主场景

第一屏就是 3D 桌面：

- 桌面材质为真实木纹或深色布面。
- 三枚铜钱位于主视觉中心。
- 相机为略俯视桌面视角，投掷中可轻微跟随但不剧烈晃动。
- UI 使用边缘 HUD，不遮挡中央投掷区。
- 结果抽屉从侧边或底部进入，中央桌面仍可见。

### 10.2 铜钱资产

铜钱使用 GLB 或 glTF 2.0：

- 真实厚度。
- 中央方孔。
- 正反面纹理不同。
- 边缘磨损、氧化、污渍和细微凹凸。
- PBR 材质包含 base color、roughness、metalness、normal。
- 物理碰撞体可以使用简化圆柱或复合碰撞体，但视觉模型必须与碰撞尺度一致。

### 10.3 光照和后处理

- 使用 HDRI 或烘焙环境光。
- 暖色侧光，桌面软阴影。
- 轻量景深、色调映射和环境遮蔽。
- 避免大面积发光阵盘、强粒子、过度模糊或遮挡铜钱的特效。

### 10.4 HUD

HUD 内容：

- 当前第几爻，共六爻。
- 问题摘要。
- 当前输入状态：准备、蓄势、释放、落定。
- 能量反馈。
- AI 设置入口。
- 结果入口。

HUD 规则：

- 文案短，避免解释性长段落。
- 移动端优先保证投掷区域不被遮挡。
- 所有按钮和输入状态必须有可触控尺寸。

## 11. 六爻与结果

领域规则保持传统三枚铜钱法：

| 总分 | 爻名 | 阴阳 | 动爻 | 变化 |
| --- | --- | --- | --- | --- |
| 6 | 老阴 | 阴 | 是 | 变阳 |
| 7 | 少阳 | 阳 | 否 | 不变 |
| 8 | 少阴 | 阴 | 否 | 不变 |
| 9 | 老阳 | 阳 | 是 | 变阴 |

六次投掷从下往上排列：

- 第 1 次为初爻。
- 第 6 次为上爻。
- 本卦由六次原始阴阳组成。
- 动爻为老阴和老阳。
- 变卦由动爻反转后得到。

结果抽屉展示：

- 用户问题。
- 本卦。
- 动爻。
- 变卦。
- 六爻排盘。
- 六次投掷记录。
- 传统依据：卦辞、象辞、动爻爻辞。
- AI 解读：headline、plainText、advice。
- 证据面板：每一爻输入摘要和落定原因。

## 12. AI 解读

第一版使用前端直连 OpenAI-compatible API：

- 用户填写 API URL、API Key 和 Model。
- 成卦后前端组装 prompt payload。
- 请求返回 JSON，格式为 `{ "headline": string, "plainText": string, "advice": string[] }`。
- JSON 校验失败时展示错误，允许重试。

AI 输入包含：

- 用户问题和问题类型。
- 六次投掷记录。
- 本卦名称、卦辞、象辞、关键词和摘要。
- 动爻爻辞、摘要和标签。
- 变卦名称、卦辞、象辞、关键词和摘要。
- 传统依据列表。

AI 行为约束：

- 只能基于给定传统依据和用户问题解读。
- 不改写经典原文。
- 不声称结果必然发生。
- 不提供高风险确定性建议。
- 输出中文 JSON，不输出 Markdown。

错误处理：

- API Key 缺失：引导打开 AI 设置。
- 请求失败：保留传统结果，展示错误原因和重试按钮。
- 返回不可解析 JSON：展示解析错误，允许重试。
- 用户取消或重置：中止请求。

## 13. 测试策略

### 13.1 单元测试

- 输入采样能生成稳定的 `InputSummary`。
- PC 钱筒不同轨迹生成不同能量、方向和角速度。
- 移动端 shake-then-still 能从 idle 到 charging 到 released。
- 弱输入会生成最小物理能量。
- `createCoinToss` 的 6、7、8、9 映射正确。
- 成卦从初爻到上爻顺序正确。
- 动爻生成变卦正确。
- AI payload 包含传统依据和投掷记录。
- AI JSON 校验能处理成功、缺字段和解析失败。

### 13.2 物理测试

- 同一个 `PhysicalTossInput` 在固定微扰种子下可复现。
- 不同输入摘要会改变初始位置、线速度或角速度。
- 正反面读取来自最终刚体 quaternion。
- `strict` 落定只在速度和面法线满足阈值时发生。
- `timeout-readable` 不调用随机正反面。
- 立边处理最终仍从物理朝向读取。
- 仿真不会让铜钱穿透桌面或飞出可见区域。

### 13.3 统计测试

在大量合成正常输入上检查分布：

- 单枚 heads/tails 接近 50/50。
- 三枚铜钱线值接近理论分布：6 为 12.5%，7 为 37.5%，8 为 37.5%，9 为 12.5%。
- 多种正常 PC 摇动模式不明显偏向某一面。
- 多种正常移动端摇晃模式不明显偏向某一爻值。

统计测试阈值应足够宽，避免因为物理仿真波动造成 flaky；同时要能捕捉明显偏差。

### 13.4 浏览器验证

- WebGL canvas 非空。
- 桌面端完整完成六次投掷。
- 移动端布局不遮挡主投掷区。
- 传感器权限允许、拒绝和不支持三种状态都有可用路径。
- 结果抽屉不遮挡中央落点。
- AI 成功、失败和重试状态可恢复。
- 低性能设备上帧率和仿真时间保持在可接受范围。

## 14. 分阶段实施

### 阶段一：物理闭环

目标：证明结果由物理落定决定。

内容：

- 定义新的 `InputSummary`、`PhysicalTossInput`、`PhysicsSnapshot` 和 `CastingEvidence`。
- 重写物理创建和落定读取接口。
- 建立严禁预生成正反面的测试。
- 保留现有领域逻辑作为结果转换入口。

完成标准：

- 任一输入路径都只能通过物理仿真获得 faces。
- 单元和物理测试覆盖落定读取和超时处理。

### 阶段二：PC 钱筒交互

目标：完成桌面端可玩的六爻主流程。

内容：

- 实现 PC 按住摇动、能量反馈、松手释放。
- 实现键盘兜底。
- 完成六次循环、进度 HUD 和证据记录。

完成标准：

- 鼠标可完成六次起卦。
- 每一爻有输入摘要和落定证据。
- 结果由领域逻辑生成。

### 阶段三：移动端摇晃

目标：完成移动端传感器投掷。

内容：

- 实现 DeviceMotion 权限流程。
- 实现 shake-then-still 检测。
- 传感器不可用时降级到触控钱筒。

完成标准：

- 支持的手机浏览器可通过摇晃完成六次起卦。
- 权限拒绝时仍可继续起卦。

### 阶段四：拟真视觉与结果体验

目标：达到博物馆桌面实拍感，并完成 AI 解读体验。

内容：

- 接入铜钱 GLB/PBR 资产。
- 调整桌面材质、灯光、阴影、相机和后处理。
- 增加音效和细节反馈。
- 重做结果抽屉、传统依据、证据面板和 AI 解读状态。

完成标准：

- 视觉中心始终是铜钱物理运动。
- 结果抽屉清晰展示传统结果和 AI 解读。
- 浏览器验证覆盖桌面和移动端关键状态。

## 15. 验收标准

- 掷钱结果不会在物理仿真前生成。
- PC 鼠标钱筒和移动端传感器都能完成六爻起卦。
- 最终 faces 来自三枚铜钱落定后的刚体朝向。
- 六爻结果与传统三枚铜钱规则一致。
- 每一爻都有可查看的投掷证据。
- AI 解读只在六爻完成后进行，失败不破坏传统结果。
- 第一屏是可交互 3D 桌面，不是营销页或说明页。
- 视觉达到真实铜钱、真实桌面和自然光照的方向。
- 测试覆盖输入映射、物理落定、六爻规则、AI payload 和浏览器主流程。

## 16. 风险与处理

- 物理仿真可能在移动端性能不足：使用简化碰撞体、固定步长和性能档位。
- 铜钱立边或长时间不落定：使用物理扰动继续解决，不直接随机判面。
- 真实物理分布可能受模型参数偏置：用统计测试校准摩擦、弹性、厚度和初始姿态范围。
- DeviceMotion 浏览器支持差异大：提供触控钱筒兜底。
- 前端直连 API Key 暴露：在 UI 中明确提示只适合个人使用，后续可迁移后端代理。
- 视觉拟真资产制作成本高：先用程序化/PBR 近似资产打通闭环，再替换高质量 GLB。

## 17. 已批准的设计决策

- 掷钱结果采用物理决定结果，不使用输入种子直接抽正反面。
- PC 端主体验采用虚拟手心或钱筒摇动后抛出。
- 移动端采用摇到足够强，然后静止自动抛出。
- AI 解读第一版采用前端直连 OpenAI-compatible API。
- 视觉方向采用博物馆级桌面实拍感。
- 当前实现可忽略；但保留可验证的六爻、卦库和 AI 适配逻辑。
- 总体架构、流程状态机、模块边界、物理可信度、视觉交互和实施验证方案均已在对话中批准。
