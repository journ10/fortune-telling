# 桌面场景、传统铜钱与手势修订规格

日期：2026-07-01
状态：已获用户口头确认，待实现

## 背景

上一轮已经把应用改成“桌面场景 + 悬浮弹窗”的主方向。用户验收截图后指出四个问题：

- 桌面场景太素，需要更有画面质感。
- 铜钱太假，看不出正反面。
- 掷钱动画像三枚钱原地起跳，不像悬空落桌。
- 手势识别没有明显摄像头启用入口。

用户补充要求：参考传统算卦常用铜钱形制来模拟。

## 参考依据

- 三枚铜钱起卦：三枚钱一次投出，六次成卦；每枚钱按正反计值，三枚合计为 6、7、8、9。
- 传统外形：采用中国方孔钱形制，即圆形、中心方孔、铜质铸造、正面钱文、背面钱背或记局文字。
- 视觉参照：以清代常见的“乾隆通宝”类方孔钱为主要视觉方向。正面模拟“乾隆通宝”四字围绕方孔排列，背面模拟清钱背面的满文/局名痕迹、氧化铜锈和磨损。

本项目只做视觉模拟，不声明生成某一枚真实古钱的精确拓片，也不改变现有随机和六爻规则。

## 目标

- 主画面保持只有桌面环境和三枚铜钱；其他输入、状态、结果仍全部走悬浮弹窗或浮窗。
- 桌面不再是纯色大平面，要有材质、光照、暗角、接触阴影和空间层次。
- 铜钱要像传统方孔圆钱，具备厚度、倒角、方孔、外郭、内郭、钱文、铜锈和正反面差异。
- 掷钱前，三枚铜钱处于桌面上方的悬浮待掷状态。
- 触发掷钱后，三枚钱从悬空状态落到桌面，不同轨迹、不同旋转、不同落点、不同反弹。
- 最终正反面必须严格服从 `tossCoins()` 已经生成的结果。
- 手势识别必须有明确摄像头启用入口和启用后的可见反馈。

## 非目标

- 不引入真实物理引擎。
- 不改变三枚铜钱起卦概率。
- 不让手势结果参与随机数生成。
- 不加入主画面固定说明、固定按钮面板或结果卡片。
- 不依赖外部生图资源作为第一版实现；先用程序化纹理保证稳定性和可控性。

## 场景设计

`TabletopScene` 继续作为常驻全屏 Three.js 场景。它需要从“纯色台面”升级为程序化桌案：

- 使用程序化 CanvasTexture 或 Three.js 纹理生成深色木纹/漆面纹理。
- 桌面加入细微色差、划痕、颗粒、暗角和中心光区。
- 光照采用暖色主光、冷色弱补光、低角度侧光和更明确的接触阴影。
- WebGL 失败时，CSS fallback 也要呈现方孔钱正反面，不退回普通圆片。

主画面仍不显示表单、卡片、侧栏、工具栏或解释文本。

## 铜钱设计

铜钱以程序化几何和贴图实现：

- 几何：圆形方孔钱，厚度约为半径的 8% 至 12%，外圆和方孔都有倒角。
- 正面：铜色偏亮，围绕方孔绘制“乾、隆、通、宝”钱文；有外郭、内郭、磨损和局部高光。
- 背面：铜色偏暗，加入青绿色氧化、砂眼、磨损和模拟满文/局名痕迹；背面纹理必须与正面一眼可分。
- 三枚铜钱可以共享基础几何，但贴图噪声和旋转角度应有轻微差异，避免像复制品。
- heads 显示正面，tails 显示背面；动画结束朝向必须与 `pendingToss.faces` 对齐。

## 动画设计

每一掷的动画分四段：

1. 悬停待掷：三枚铜钱停在桌面上方，带轻微上下浮动和投影。
2. 下落抛掷：触发后从不同高度和水平位置开始落向桌面，中途有轻微前后漂移。
3. 翻滚反弹：每枚钱有独立角速度，触桌后出现短促反弹、滑动和阻尼。
4. 停稳保留：最终停在桌面上，短暂停留后再回到下一掷的悬浮待掷状态。

动画时长建议从当前约 320ms 提升到 1500ms 至 1900ms。`prefers-reduced-motion` 下可缩短位移和旋转，但仍要有“落下并停稳”的状态变化。

## 手势设计

手势识别恢复为悬浮交互，不放入主画面固定面板：

- 起卦开始后，显示一次悬浮手势启用窗，提供“启用摄像头”和“手动投掷”。
- 用户启用后，请求摄像头权限，并显示小型浮窗：摄像头预览、当前识别状态、握拳后张开手的触发提示。
- 用户跳过时，手势窗隐藏，仍可点击/键盘投掷。
- 摄像头失败、权限拒绝或 MediaPipe 加载失败时，只在浮窗内显示错误和重试入口。
- 手势触发仍调用现有 `onTossRequest()`，不绕过 `pendingToss` 防重复逻辑。

## 测试要求

- `TabletopScene` fallback 下仍能完成 pending toss settle，settle 时长跟随新动画时间。
- `TabletopScene` 的 DOM 标记或 fallback 能区分 heads/tails。
- 新的手势浮窗能渲染启用入口。
- 点击“手动投掷”会隐藏手势浮窗。
- 点击“启用摄像头”会调用摄像头启动函数；失败时渲染错误。
- App 起卦后应出现手势启用入口，但不阻塞手动投掷路径。
- 现有 AI-only 解读、悬浮弹窗、原始卦象保留逻辑不得回退。

## 验收标准

- 桌面场景肉眼看起来不是纯色背景，具备材质和光影层次。
- 三枚钱掷前悬空，掷后落桌，不再原地同步起跳。
- 正反面从视觉上可区分，正面接近“乾隆通宝”方孔钱，背面有不同纹样和铜锈。
- 每枚钱的轨迹、旋转和落点不完全一致。
- 摄像头手势有明确启用入口，启用后能看到预览或状态浮窗。
- 无论点击还是手势触发，六次投掷记录和最终卦象仍由现有规则生成。
